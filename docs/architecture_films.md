# Архитектура: фильмы и Топ 250

## Принцип

**В таблице `movies` хранятся ВСЕ фильмы** — единственный источник правды о фильме (название, год, постер, описание, жанры и т.д.). Ни один фильм не существует только «в Топ 250»: если он в Топ 250, он обязательно есть в `movies`.

**Таблица `kinopoisk_top250`** — только связь «этот фильм входит в текущий список Топ 250 и на такой-то позиции». Хранит:
- `movie_id` (FK → `movies.id`) — ссылка на фильм;
- `position` — место в списке (1–250);
- при необходимости `kinopoisk_id` и `updated_at` для удобства.

Данные о названии, постере, жанрах и т.д. всегда берутся из `movies`.

## Откуда фильмы попадают в `movies`

1. **Обычный подбор** — ИИ рекомендует фильмы → для каждого вызывается Кинопоиск (или кэш) → запись создаётся/обновляется в `movies`.
2. **Избранное / «Посмотрел»** — при добавлении фильма вызывается Кинопоиск (или кэш) → запись в `movies`.
3. **Синхронизация Топ 250** — при загрузке списка с API для каждого из 250 фильмов:
   - полные данные сохраняются в `movies` (по `kinopoisk_id`);
   - в `kinopoisk_top250` пишется только `(movie_id, position)`.

Таким образом, после синхронизации Топ 250 в `movies` есть как минимум 250 фильмов, а в выгрузке `/export_films` они все присутствуют.

## Схема (реализовано)

- **movies** — как и раньше: id, kinopoisk_id, title, year, age_rating, rating_kp, poster_url, description, genres, countries, votes, raw_json, updated_at. Сюда пишутся все фильмы.
- **kinopoisk_top250** — id, **movie_id** (FK → movies.id), kinopoisk_id, position, title, year, genres, … (поля для обратной совместиости), updated_at. При синхронизации сначала создаётся/обновляется запись в `movies`, затем в top250 сохраняются `movie_id` и `position`. При выборке (подбор, экспорт) данные о фильме берутся из `movies` через JOIN по `movie_id`.

## Дублирование и отчёты

**В таблицах:** В `kinopoisk_top250` до сих пор есть колонки title, year, poster_url и т.д. (исторически и для совместиости). При синке они заполняются — то есть **данные о фильме физически дублируются**: и в `movies`, и в `kinopoisk_top250`. Один и тот же фильм в Топ 250 хранится в `movies` один раз и ещё раз «отражён» в строке top250.

**При отчётах и в боте** дублирование не используется как два разных источника:
- **Подбор в боте (Топ 250)** — данные собираются из **двух таблиц через JOIN**: список позиций из `kinopoisk_top250`, названия/постеры/жанры из `movies` по `movie_id`. Используется один источник правды — `movies`.
- **Выгрузка `/export_films`** — только таблица **`movies`** (все фильмы, в т.ч. из Топ 250).
- **Выгрузка `/export_top250`** — список позиций из `kinopoisk_top250`, а **данные о фильмах (название, год, постер и т.д.) берутся из `movies` через JOIN**. То есть при отчёте данные снова собираются из двух таблиц, но источник правды для полей фильма — только `movies`.

Итого: один и тот же фильм в БД есть в `movies` один раз и может быть «продублирован» полями в строке `kinopoisk_top250`. В отчётах и в боте мы не смешиваем два варианта одних и тех же полей — для названия, постера и т.д. везде используется `movies` (напрямую или через JOIN).

## Вариант с атрибутом в одной таблице

Альтернатива: одна таблица `movies` и флаги `in_top250`, `top250_position`. Тогда Топ 250 — просто фильтр/сортировка по полям в `movies`. Мы выбрали вариант с отдельной таблицей-списком (`kinopoisk_top250`), чтобы не раздувать `movies` и хранить только актуальный список позиций (1–250) без лишних полей.
