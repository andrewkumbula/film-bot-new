# Маппинг и очистка дубликатов фильмов (пустые ↔ с данными Кинопоиска)

## Источник данных

- **Таблица с фильмами:** в проекте это `movies` (в разговоре — «films»).
- **Пустая запись:** запись в `movies`, у которой нет данных Кинопоиска: нет `kinopoisk_id` или нет полезных полей (poster_url, rating_kp, age_rating и т.д.).
- **Полная (главная) запись:** запись в `movies` с заполненными данными Кинопоиска; на неё будем переводить ссылки при слиянии.

## Если пустая запись уже используется

При маппинге «пустая = главная» пустая запись может быть связана из других таблиц по `movie_id`. Два подхода:

### Вариант A: Мягкое удаление

- Пустую запись **не удаляем**, а помечаем: например колонка `merged_into_movie_id` (FK на `movies.id`) или `is_deprecated = 1`.
- В логике приложения: при выборке фильмов по `movie_id` подставлять данные из главной записи, если у текущей стоит `merged_into_movie_id` (или не показывать deprecated в выдачах, но ссылки в `favorites`/`watched`/и т.д. остаются на старый id).
- **Плюсы:** не трогаем ссылки в других таблицах, откат простой.  
- **Минусы:** нужна миграция (новая колонка), вся логика чтения фильмов должна учитывать «подстановку» из главной записи.

### Вариант B: Жёсткое удаление с переносом ссылок (рекомендуется)

- Во **всех** таблицах, где есть `movie_id`, заменить ссылку с id пустой записи на id главной:  
  `UPDATE ... SET movie_id = <id_главной> WHERE movie_id = <id_пустой>`.
- Учесть уникальность: в `favorites`/`watched`/`not_interested` есть `UNIQUE(user_id, movie_id)`. Если у пользователя уже есть главная запись в избранном, а мы переводим ссылку с пустой на главную — получится дубликат (user_id, main_id). Нужно: либо оставить одну строку (удалить дубликат после UPDATE), либо не выполнять UPDATE для такой строки и удалить строку, ссылающуюся на пустую (т.к. главная уже в избранном).
- После переноса ссылок выполнить `DELETE FROM movies WHERE id = <id_пустой>`.
- **Плюсы:** одна источник правды, нет «мёртвых» записей.  
- **Минусы:** нужно аккуратно обработать уникальные ограничения и порядок операций.

## Таблицы, где хранится movie_id

| Таблица             | Ссылка     | Ограничение              |
|---------------------|------------|---------------------------|
| `favorites`         | `movie_id` | UNIQUE(user_id, movie_id) |
| `watched`           | `movie_id` | UNIQUE(user_id, movie_id) |
| `not_interested`    | `movie_id` | UNIQUE(user_id, movie_id) |
| `kinopoisk_top250`  | `movie_id` | —                         |

При переносе ссылок (вариант B) для каждой из этих таблиц:

1. Сделать `UPDATE table SET movie_id = main_id WHERE movie_id = empty_id`.
2. В `favorites`, `watched`, `not_interested`: после UPDATE могут появиться дубликаты по (user_id, movie_id). Удалить дубликаты, оставив одну запись (например по `id` или `created_at`), затем при необходимости почистить оставшиеся строки, ссылающиеся на `empty_id`, если они стали лишними.
3. В `kinopoisk_top250`: обычно у строки top250 уже стоит главная запись; если вдруг на пустую ссылается — после UPDATE одна строка top250 будет указывать на главную (дублей по смыслу не будет, т.к. top250 хранит позиции по kinopoisk_id).
4. В конце: `DELETE FROM movies WHERE id = empty_id`.

## Реализация (уровень 1)

- **Сервис:** `app/services/movie_mapping_cleanup.py` — выборка пустых/полных записей из `movies`, нормализация названия, поиск пары «100% название + год ±1», перенос ссылок и удаление пустой записи (вариант B).
- **Планировщик:** раз в сутки в **02:30** вызывается `movie_mapping_cleanup_scheduler` → `run_cleanup_level1()`.
- **Ручной запуск:** `python scripts/run_movie_cleanup.py` из корня проекта (для проверки без ожидания ночи).

Уровни 2 и 3 (ИИ при разнице года > 1 и при нечётком совпадении названия) пока не реализованы.

## Резюме

- Пустые и полные записи берём из таблицы **movies** (films).
- Пустая = нет данных Кинопоиска (нет `kinopoisk_id`).
- Реализовано **жёсткое удаление** с предварительной заменой `movie_id` на id главной записи во всех таблицах: `favorites`, `watched`, `not_interested`, `kinopoisk_top250`, с удалением дубликатов по (user_id, movie_id) после UPDATE.
