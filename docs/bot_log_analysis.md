## 1. Ошибки

### 1.1 UnboundLocalError в «Избранное» (критично)

**Что в логе:** При нажатии кнопки «⭐️ Избранное» в главном меню:

```
UnboundLocalError: cannot access local variable 'settings' where it is not associated with a value
  File ".../app/routers/favorites.py", line 17 (позже 25), in show_favorites
    favorites = await list_favorites_for_user(settings, message.from_user.id, limit=10)
```

**Причина:** В обработчике `show_favorites` переменная `settings` берётся из замыкания `get_router(settings)`. Если где-то в этой же функции есть присваивание в `settings` (например `settings = load_settings()` в другой ветке или в цикле), Python считает `settings` локальной на всю функцию, и до присваивания к ней обращаться нельзя.

**Что сделано в коде:** В начале `show_favorites` добавлена явная привязка `s = settings` и дальше везде используется `s`. Так замыкание гарантированно не конфликтует с локальными переменными.

**На проде:** Убедиться, что задеплоена версия с этим фиксом и перезапустить бота.

### 1.2 Kinopoisk API — 403 Forbidden (массово)

**Что в логе:** Почти все запросы к `api.poiskkino.dev/v1.4/movie/search` возвращают:

```
Kinopoisk API search: status 403 for query=Шрек 2001
Kinopoisk API search: status 403 for query=Кун-Фу Панда 2008
...
```

**Вывод:** API Кинопоиска (poiskkino.dev) отклоняет запросы: 403 обычно означает запрет доступа (лимит, неверный/просроченный ключ, блокировка по IP и т.п.).

**Рекомендации:**
- Проверить в @poiskkinodev_bot актуальность токена и лимиты.
- Убедиться, что с сервера, где крутится бот, не блокируют доступ к API.
- Пока 403 — подбор фильмов работает только на данных ИИ (без постеров/рейтингов Кинопоиска); после восстановления доступа карточки снова начнут обогащаться.

## 2. Оптимизация флоу подбора фильмов (по метрикам movie_flow_timing)

Типичные замеры одного полного прогона (обычный подбор):

| Этап | Время (примеры из лога) | Доля от total |
|------|-------------------------|----------------|
| **llm** | 10.69 s, 12.70 s, 12.45 s, 11.64 s | **~35–45%** |
| **kinopoisk_gather** | 16.85 s, 6.98 s, 23.42 s, 1.90 s | **~10–65%** |
| **build_and_send_cards** | 0.41–0.88 s | ~2–3% |
| **filter_rating / filter_ni_watched / filter_recently** | &lt; 0.1 s | &lt;1% |
| **movie_id_resolve_and_state** | 0.02–0.03 s | &lt;1% |
| **top250_positions** | ~0 s | — |
| **total** | **~20–37 s** (Duration 14–37 s в логе) | 100% |

### Что больше всего занимает времени

1. **LLM (запрос рекомендаций)** — стабильно **~10–12.5 сек** на один запрос. Это основной и самый предсказуемый узкий участок.
2. **kinopoisk_gather** — **~2–23 сек** в зависимости от ответов API:
   - При 403 все запросы всё равно выполняются по одному (семафор + пауза), поэтому 15 фильмов дают ~16+ сек ожидания.
   - Когда API отвечает 200, этап может занимать 2–7 сек; при таймаутах — до 23 сек.

Итого: основное время пользователя (~25–37 с до ответа) складывается из **одного вызова LLM** и **множества последовательных запросов к Кинопоиску** (по одному на кандидата, с задержкой между запросами).

### Рекомендации по оптимизации

- **LLM:**
  - Рассмотреть более быструю модель для сценария подбора (если в конфиге можно переключить).
  - Кэшировать рекомендации по хешу предпочтений (жанры, возраст, настроение и т.д.), где это уместно.
- **Кинопоиск:**
  - После восстановления 200: оставить семафор (чтобы не ловить 429), но по возможности сократить паузу между запросами в пределах правил API.
  - Кандидатов после LLM много (14–15); фильтрация и показ 5 карточек. Имеет смысл сначала отфильтровать/ограничить список (например, по рейтингу из БД), а потом запрашивать Кинопоиск только для оставшихся (например, для 7–8), чтобы сократить число запросов и время kinopoisk_gather.
- **Карточки:** build_and_send_cards уже доли секунды; приоритет низкий.
